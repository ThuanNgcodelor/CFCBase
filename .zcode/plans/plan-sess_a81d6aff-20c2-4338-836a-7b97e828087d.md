# Kế hoạch triển khai PWA Push Notification cho BookingBase

## Quyết định đã chốt
- **SW**: chuyển `vite-plugin-pwa` sang **`injectManifest`** + custom `frontend/src/sw.js` (giữ Workbox precache + thêm handler).
- **VAPID public key**: frontend **fetch từ backend** `GET /api/v1/push/vapid-public-key`.
- **UX bật thông báo**: **Trang Notifications** + **Trang Profile**.
- **Scheduler cleanup**: **bỏ qua MVP**, chỉ xử lý 404/410 real-time.
- Các rules ở phần trước: push cho mọi loại notification, recipient theo logic hiện có, logout → unsubscribe + xóa record, login lại → re-subscribe ngầm.

---

## Phase 0 — Audit & chuẩn bị môi trường (không code app)

**Mục tiêu**: sinh VAPID keypair, verify HTTPS/SW hiện tại.

**Việc làm**:
1. Sinh cặp VAPID key bằng công cụ (ví dụ `npx web-push generate-vapid-keys` hoặc dùng tool của thư viện Java). Lưu:
   - **Private key** + **subject** `https://cfcbooking.io.vn` → chuẩn bị đưa vào backend env / `application.yml`.
   - **Public key** (base64url) → backend trả qua endpoint (không hardcode FE).
2. Verify `https://cfcbooking.io.vn` reachable, manifest fetch được, SW register được (DevTools → Application).

**File**: không sửa file app. Chỉ ghi nhận key vào nơi an toàn (env server, KHÔNG commit).

**Test manual**: Lighthouse PWA audit; cài PWA trên Chrome Android + Safari iOS 16.4+; check `start_url`/icon.

**Hoàn thành**: có keypair sẵn sàng; PWA cài được.

## Phase 1 — PWA / Service Worker nền tảng

**Mục tiêu**: chuyển `generateSW` → `injectManifest`, tạo `sw.js` có skeleton `push`/`notificationclick` (chưa subscribe).

**File sửa/tạo**:
1. **`frontend/vite.config.js`** — đổi khối `VitePWA`:
   - `strategies: 'injectManifest'`
   - `srcDir: 'src'`, `filename: 'sw.js'`
   - Giữ `registerType: 'autoUpdate'`, `manifest`, `includeAssets`, `devOptions`.
   - Xóa comment stale về "injectManifest"/"VAPID thuần" cho đúng thực tế.
2. **`frontend/src/sw.js`** (MỚI) — Workbox `precacheAndRoute` (injectManifest tự inject manifest precache) + handler placeholder:
   - `self.addEventListener('push', ...)` skeleton.
   - `self.addEventListener('notificationclick', ...)` skeleton.
   - `self.addEventListener('install'/'activate')` cho `skipWaiting`/`clients.claim` (tương thích `autoUpdate`).

**Lý do**: `generateSW` không cho custom event handler; `injectManifest` cho phép viết SW riêng但仍 precache qua Workbox.

**Rủi ro**: mất precache nếu config sai → phải test kỹ; SW cũ không update → cần unregister + hard refresh khi test.

**Test manual**: `npm run build && npm run preview` → DevTools Application → SW mới active, precache list đầy đủ; app vẫn chạy offline; unregister SW → reload → SW cài lại.

**Hoàn thành**: SW injectManifest active, precache OK, app chạy bình thường (chưa có push thực).

## Phase 2 — Backend: Push Subscription API + Web Push sender

**Mục tiêu**: entity/repo/dto/service/controller cho subscription + Web Push sender + endpoint VAPID public key.

**File tạo/sửa backend**:
1. **`backend/pom.xml`** — thêm dependency Web Push Java (lib `nl.martijndwars:webpush` phiên bản tương thích Java 21, + BouncyCastle nếu lib yêu cầu).
2. **`backend/src/main/resources/application.yml`** — thêm khối:
   ```yaml
   app:
     webpush:
       public-key: ${VAPID_PUBLIC_KEY:}
       private-key: ${VAPID_PRIVATE_KEY:}
       subject: https://cfcbooking.io.vn
   ```
   (giữ nguyên phần `app.cors` hiện có).
3. **`entity/PushSubscription.java`** (MỚI) — theo convention `Notification.java`:
   - `@Getter @Setter @Entity @Table(name="push_subscriptions")`.
   - `@Id @GeneratedValue(UUID) String id`.
   - `@ManyToOne User user` (FK `user_id`, nullable=false).
   - `endpoint` (length 500, nullable=false), `p256dhKey`, `authKey`.
   - `expirationTime` (LocalDateTime, nullable).
   - `deviceType`, `userAgent` (nullable).
   - `isActive` (boolean, default true), `createdAt`, `lastSeenAt`, `revokedAt`.
   - `@Table(indexes=[(user_id, is_active)], uniqueConstraints=[(endpoint)])`.
4. **`repository/PushSubscriptionRepository.java`** (MỚI) — `JpaRepository<PushSubscription, String>` + query:
   - `List<PushSubscription> findByUserIdAndIsActiveTrue(String userId)`.
   - `Optional<PushSubscription> findByEndpoint(String endpoint)`.
5. **`dto/PushSubscriptionRequest.java`** + **`dto/PushSubscriptionResponse.java`** (MỚI) — `@Getter @Builder`, theo convention `NotificationResponse`.
6. **`config/WebPushConfig.java`** (MỚI) — `@Configuration`, bean `PushService` (hoặc wrapper) cấu hình VAPID từ `@Value("${app.webpush...}")`.
7. **`service/PushService.java`** (MỚI) — `@Service @RequiredArgsConstructor @Slf4j`:
   - `sendPush(PushSubscription sub, Object payload)` — gọi thư viện webpush; catch exception; phân loại HTTP status 404/410 → gọi `PushSubscriptionService.deactivate(sub)`; 413 → log; không throw.
   - `getVapidPublicKey()` — trả public key cho controller.
8. **`service/PushSubscriptionService.java`** (MỚI) — `@Service @RequiredArgsConstructor`:
   - `@Transactional subscribe(userId, PushSubscriptionRequest)` — UPSERT theo `endpoint`: nếu endpoint tồn tại → cập nhật keys/user/isActive=true; else insert mới.
   - `@Transactional unsubscribe(userId, endpoint)` — deactivate + set `revokedAt` (dùng cho logout).
   - `@Transactional deactivate(PushSubscription)` — `isActive=false` (dùng khi 404/410).
   - `findActiveByUser(userId)`.
9. **`controller/PushSubscriptionController.java`** (MỚI) — `@RestController @RequestMapping("/api/v1/push") @RequiredArgsConstructor`, theo convention `NotificationController`:
   - `GET /vapid-public-key` → `Map.of("publicKey", ...)` (**permitAll** hoặc authenticated — chọn authenticated cho gọn, dùng `@AuthenticationPrincipal`).
   - `POST /subscriptions` (body `PushSubscriptionRequest`) → subscribe, trả `PushSubscriptionResponse`.
   - `DELETE /subscriptions` (body `{endpoint}` hoặc query) → unsubscribe.
   - `GET /subscriptions` (optional) → list subscription của user (cho UI "thiết bị đã đăng ký").

**Rủi ro**: VAPID key sai → push fail; payload > 4KB → 413; lib tương thích Boot 4 / Java 21 (lib độc lập Spring nên OK, test kỹ).

**Test manual**: Postman/curl `GET /vapid-public-key` trả key; `POST /subscriptions` với subscription mock → DB có row; gọi `sendPush` với subscription mock → push tới FCM/APNs test endpoint.

**Hoàn thành**: API subscribe/unsubscribe hoạt động; sender gửi được push test.

## Phase 3 — Backend: tích hợp Push vào NotificationEventListener

**Mục tiêu**: thêm nhánh `sendPushIfSubscribed` vào orchestrator hiện có, **không đổi** WS/email/DB.

**File sửa backend**:
1. **`event/NotificationEventListener.java`** — inject `PushService` + `PushSubscriptionService`. Trong `handle()`, **sau** khối email, thêm khối try/catch thứ 3 (cùng pattern):
   ```java
   try {
       sendPushIfSubscribed(event, payload);  // payload = NotificationResponse từ createNotification
   } catch (Exception ex) {
       log.error("Failed to send push for recipient {}", event.recipientId(), ex);
   }
   ```
   - `sendPushIfSubscribed`: nếu `payload != null` → query `findActiveByUser(recipientId)` → gửi push tới từng subscription. Payload chứa `{title, message, targetUrl, sourceType, sourceId, type}` (đủ để SW dựng deep link). Update `lastSeenAt` khi gửi thành công.
   - **Quan trọng**: chỉ gửi push khi `payload != null` (tức createNotification không bị idempotent skip) → tránh gửi push cho notification đã tồn tại.

**Rủi ro**: nếu push chậm → block listener (listener hiện đã synchronous). Mitigation: Web Push gửi nhanh (HTTP POST tới FCM); nếu lo → có thể bọc `@Async` cho nhánh push (nhưng cần test transaction context). **MVP: giữ synchronous theo pattern hiện có** để nhất quán.

**Test manual**: tạo booking với user đã subscribe (giả lập qua Phase 4) → verify push nhận được; tạo lại cùng booking (idempotent) → không có push thứ 2; simulate endpoint 404 → subscription bị deactivate, WS/email vẫn chạy.

**Hoàn thành**: tạo booking/approve/reject/cancel → recipient nhận push nếu có subscription active.

## Phase 4 — Frontend: hook subscribe + API + UX bật thông báo

**Mục tiêu**: xin quyền, subscribe, gửi lên backend, UX bật/tắt trên trang Notifications + Profile.

**File tạo/sửa frontend**:
1. **`frontend/src/api/pushApi.js`** (MỚI) — theo convention `notificationApi.js`:
   - `getVapidPublicKey()` → `GET /push/vapid-public-key` (dùng `baseApi`).
   - `subscribe(body)` → `POST /push/subscriptions`.
   - `unsubscribe(endpoint)` → `DELETE /push/subscriptions`.
   - `listSubscriptions()` → `GET /push/subscriptions` (optional).
2. **`frontend/src/hooks/usePushNotifications.js`** (MỚI) — hook quản lý trạng thái:
   - Detect support: `'serviceWorker' in navigator && 'PushManager' in window`.
   - `enableNotifications()`: `Notification.requestPermission()` → nếu granted → `navigator.serviceWorker.ready` → `registration.pushManager.subscribe({userVisibleOnly:true, applicationServerKey: urlBase64ToUint8Array(publicKey)})` → `pushApi.subscribe({endpoint, p256dh, auth, expirationTime, deviceType, userAgent})`.
   - `disableNotifications()`: get current subscription → `subscription.unsubscribe()` → `pushApi.unsubscribe(endpoint)`.
   - State: `supported`, `permission` (`default`/`granted`/`denied`), `subscribed`.
   - Helper `urlBase64ToUint8Array` (base64url decode VAPID public key).
3. **`frontend/src/pages/Notifications.jsx`** — thêm card "Thông báo đẩy" đầu trang:
   - Nút "Bật thông báo" / "Tắt thông báo" / trạng thái "Đã bật" / "Bị chặn" (khi permission=denied, ẩn nút, hiện gợi ý vào Settings).
4. **`frontend/src/pages/Profile.jsx`** — thêm section cài đặt tương tự (dùng cùng hook).
5. **iOS detect**: trong hook, detect standalone (`window.matchMedia('(display-mode: standalone)').matches` hoặc `navigator.standalone`) → nếu iOS mà KHÔNG standalone → hiện hướng dẫn "Thêm vào màn hình chính".

**Rủi ro**: `requestPermission` quá sớm → deny vĩnh viễn (mitigate: chỉ gọi khi user click nút, không auto). iOS cần standalone.

**Test manual**: click "Bật" → permission granted → DevTools Application → Push Messaging thấy subscription; Network thấy POST `/push/subscriptions`; test DevTools "Emulate push" → SW log nhận; "Tắt" → subscription gone.

**Hoàn thành**: user bật/tắt được; subscription lưu backend; re-subscribe ngầm khi login lại (hook chạy trên mount của DashboardLayout, check permission granted + chưa subscribe → tự subscribe).

## Phase 5 — Frontend: Service Worker push + notificationclick + deep link

**Mục tiêu**: SW xử lý `push` event → showNotification; `notificationclick` → focus/open + navigate.

**File sửa frontend**:
1. **`frontend/src/sw.js`** — implement đầy đủ handler:
   - **`push` event**: parse `event.data.json()` → `{title, message, targetUrl, sourceType, sourceId}`. Trước khi show, check `clients.matchAll({type:'window', includeUncontrolled:true})` — nếu có client **focused** (`client.visibilityState==='visible'` && cùng origin) → **không show** (tránh duplicate với WS toast). Else `self.registration.showNotification(title, {body, icon:'/icons/icon-192.png', badge:'/icons/icon-72x72.png', data:{url}})`. `event.waitUntil(...)`.
   - **`notificationclick` event**: `event.notification.close()`. Dùng `clients.matchAll` tìm client đang mở cùng origin → nếu có → `client.focus()` + `client.postMessage({type:'NAVIGATE', url})` (app lắng nghe → `navigate(url)`). Nếu không → `clients.openWindow(url)`. `event.waitUntil(...)`.
   - **`message` listener** (app → SW): để app báo "đang focus" nếu cần sync trạng thái (optional).
2. **App lắng nghe navigate message**: thêm trong `NotificationContext` hoặc `DashboardLayout` — `navigator.serviceWorker.addEventListener('message', ...)` → nếu `data.type==='NAVIGATE'` → `navigate(data.url)`.
3. **Deep link URL**: dựa trên payload `sourceType`/`sourceId`/`targetUrl`:
   - `BOOKING_ROOM` + admin recipient → `/admin/approvals/{sourceId}`.
   - `BOOKING_CAR` tương tự.
   - Employee recipient → `/rooms` hoặc `/cars` (theo `targetUrl` hiện có).
   - SW chỉ cần dùng `targetUrl` nếu có, else fallback heuristic theo `sourceType`.

**Rủi ro**: iOS mở PWA fresh thay vì focus (giới hạn nền tảng); deep link sai màn hình → test kỹ mapping.

**Test manual**: app đóng → push → click → mở đúng `/admin/approvals/:id`; app mở focus → push không show (WS toast lo); app mở không focus → push show + click → focus + navigate.

**Hoàn thành**: click push mở đúng màn hình; không duplicate khi focus.

## Phase 6 — Logout/Login lifecycle

**Mục tiêu**: logout → unsubscribe + xóa record; login → re-subscribe ngầm.

**File sửa frontend**:
1. **`frontend/src/api/authApi.js`** — trong `logout()` (line 46-58), **trước** khi remove cookies, thêm:
   - Lấy current push subscription qua `navigator.serviceWorker.ready → pushManager.getSubscription()`.
   - Nếu có → `subscription.unsubscribe()` + `pushApi.unsubscribe(endpoint)` (best-effort, try/catch — không block logout nếu fail).
2. **Re-subscribe ngầm khi login**: trong `usePushNotifications` hook, chạy trên mount (trong `DashboardLayout`): nếu `supported && permission==='granted' && !subscribed` → tự gọi `enableNotifications()` ngầm (không hỏi lại quyền vì đã granted). Nếu `permission==='default'` → KHÔNG tự hỏi (chỉ đợi user click nút).

**Rủi ro**: logout fail unsubscribe → thiết bị vẫn nhận (mitigate: backend cũng nên cleanup khi nhận 404/410 sau này; MVP chấp nhận best-effort). Re-subscribe race condition với WS connect (khác cơ chế, không conflict).

**Test manual**: logout → verify subscription gone (DevTools + DB); login lại → verify re-subscribe ngầm (không có dialog xin quyền); đổi tài khoản A→B trên cùng thiết bị → B nhận push của B, không nhận của A.

**Hoàn thành**: lifecycle sạch, đúng rule đã chốt.

## Thứ tự thực thi

## Không làm trong scope này
- Flyway/Liquibase (giữ ddl-auto).
- Scheduler cleanup (phase sau).
- Reminder (`BOOKING_REMINDER`).
- Thay đổi recipient logic, WS auth, email rule.
